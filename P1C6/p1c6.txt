=Templates=

    Compile-time polymorphism is achieved in C++ through templates.
    
    -What is a template?-
    
        - Class or function with template parameters.
        - Params can stand in for any type.
        - Compiler will stamp out a bespoke template instantiation when a template is used with a type.


    -What is Template Instantiation?-
    
        - The Process of creating a class or a function from a template.
        - Template instantiations are sometimes called concrete classes and conrete types.
    
    -What are templates for?-
 
        - Instead of having redundant code, a single template may be written. The compiler then generates new template instances whenever a new combination of types in the template parameters is encountered.


=Declaring Templates=

    -How are templates declared?-
 
        - With a template prefix consisting of the word "template" wrapped in angle brackets <>.
        - One or more template paramters are placed inside the angle brackets.
        - Parameters are declared using either the typename or class keywords followed by an identifier.
        - template<typename T> declares that the template takes a template parameter T.


=Template Class Definitions=

    Example:
        ```
        template<typename X, typename Y,, typename Z>
        struct MytemplateClass {
            X foo(Y&);
        private:
            Z* member;
        }
        ```

    Description:
        - "template" keyword begins the template prefix.
        - Declarations of one or more template parameters go inside the angle brackets.
        - "typename" or "class" keywords are interchangable for declarig template parameters, followed by an identifier.
        - Within MyTemplateClass, x, y, and z are used as if they were any fully specified type such as int or a user-defined class.
        - The foo method takes a Y reference, returns an X.
        - Members can be declared that include template parameters, such as a pointer to Z as in the example.
        - Apart from the prefix beginning, this template class is almost identical to a non-template class.

-Template Function Definitions-

    Template functions can be specified, and they can take template parameters. (Listing 6_2.cpp)

-Instantiating Templates-

    How do you instantiate a template class?

        ```
        tc_name<t_param1, t_param2, ...>my_concrete_class{ ... };
        ```

    Description:
        - tc_name: Template class' name.
        - t_param1, t_param2: Template parameters.
        - Use any initialization syntax you prefer as if it were a normal type.
    
    Instantiating a Template Function is similar:

        ```
        auto result = tf_name<t_param1, t_param2, ...>(f_param1, f_param2, ...);
        ```

        - tf_name: Template Function's name.
        - Fill in params as if it were a normal type.
        - Template function instantiation invoked with parentheses and function parameters. 


=Named Conversion Functions=
    What are they?
    Language features that explicitly convert one type into another type.

    When are they used?
    Sparingly, in situations where implicit conversions or constructors cannot get the types that are required.

    How do they work?
    Accepting a single object parameter, the object that is to be cast, and a single type parameter (the parameter you are casting to).

    Modifying a const object:
        - const_cast required to cast away "const" qualifier.
    
    Similarity to Template Functions:
    Conceptually very close, but technically not Template Functions.

    -const_cast-
 
    Removes const modifier so that const values can be modified.

    ```
    // Listing 6-3

    // encased_solo parameter is a const reference, any attempt to modify it would result in a compiler error. However, by using const_cast, we can remove the const qualifier and modify the value.

    #include <cstdio>

    void carbon_thaw(const int& encased_solo) {
        // encased_solo++; // error: increment of read-only reference 

        // Use const_cast to remove the const qualifier and modify the value:
        auto& hibernation_sick_solo = const_cast<int&>(encased_solo);
        hibernation_sick_solo++;
    }
    ```

    -static_cast-
    
    Reverses a well-defined implicit conversion, for example: An integer type to another integer type.

    The object-to-cast is of some type that the desired-type implicitly converts to.

    Why might you need static_cast?
    Implicit casts generally are not reversible.

    ```
    // This program defines an increment_as_short function that takes a void pointer, casts it to a short pointer, increments the value it points to, and returns the incremented value. The main function initializes a short variable with the value 665, calls the increment_as_short function with its address, and prints the result.

    // Some low-level applications might require you to interpret raw bytes as an integer type

    #include <cstdio>

    // target parameter is a void pointer.
    short increment_as_short(void* target) {
        // Use static_cast to convert the void pointer to a short pointer, then dereference it to get the value, increment it, and return the result.
        auto as_short = static_cast<short*>(target);
        *as_short = *as_short + 1;
        return *as_short;
    }

    int main() {
        short beast{ 665 };
        auto mark_of_the_beast = increment_as_short(&beast);
        printf("%d is the mark_of_the_beast.", mark_of_the_beast);
    }

    // The implicit conversion of short* to void* is well defined. Attempting ill-defined conversions with static_cast, such as converting char* to a float* will result in a compile-time error.
    ```

    -reinterpret_cast-
    
    Low-level programming sometimes requires that you perform type conversions that are not well defined.

    Complete control over memory is often a requirement of system and embedded programming.

    ```
    // This program demonstrates how to use a pointer to access a memory-mapped timer. It casts the address 0x1000 to a pointer to an unsigned long and prints the value at that address.

    #include <cstdio>

    int main() {
        // reinterpret_cast takes a type parameter that corresponds to the desired pointer type. In this case, we want to treat the address 0x1000 as a pointer to an unsigned long.
        auto timer = reinterpret_cast<const unsigned long*>(0x1000);
        printf("Timer is %lu.", *timer);
    }
    ```

    The compiler will not know whether the memory at address 0x1000 contains an unsigned long, it is the programmer's responsibility to ensure correctness.

    The compiler forces you to employ reinterpret_cast, because the programmer takes full responsibility.

    -narrow_cast-
    
    Listing 6-6 shows a custom static_cast that performs a runtime check for narrowing (a loss of information).

    When converting from an int to a short, if the value of int fits in to the short, the conversion is reversible and no narrowing occurs.

    If the value of int is too big for the short, the conversion isn't reversible and will result in narrowing.
    
    ```
    // Listing 6-7

    // A program using narrow_cast to safely convert an int to a short. The first conversion succeeds because 496 can be represented as a short, while the second conversion throws a runtime error because 142857 cannot be represented as a short without loss of information.

    #include <cstdio>
    #include <stdexcept>

    // From Listing 6-6.
    template <typename To, typename From>
    To narrow_cast(From value) {
        const auto converted = static_cast<To>(value);
        const auto backwards = static_cast<From>(converted);
        // Runtime check for narrowing.
        if(value != backwards)
            throw std::runtime_error{ "Narrowed!" };
        return converted;
    }

    int main() {
        int perfect{ 496 };
        // perfect is narrow_cast to a short called perfect_short. Since 496 can be represented as a short, the conversion succeeds and perfect_short is assigned the value 496.
        const auto perfect_short = narrow_cast<short>(perfect);
        printf("perfect_short: %d\n", perfect_short);
        try {
            int cyclic{ 142857 };
            // Attempt to narrow_cast cyclic to the short cyclic_short. Since 142857 cannot be represented as a short without loss of information, the narrow_cast function throws a runtime error, which is caught by the catch block, and an error message is printed.
            const auto cyclic_short = narrow_cast<short>(cyclic);
            printf("cyclic_short: %d\n", cyclic_short);
        } catch(const std::runtime_error& e) {
            printf("Exception: %s\n", e.what());
        }
    }
    ```

=SimpleUniquePointer: A Teplate Class Example=

What is a "unique pointer"?

An RAII (Resource Acquisistion Is Initialization) wrapper around a free-store-allocated object.

    - The unique pointer has a single owner at a time.
    - When a unique pointer's lifetime ends, the pointed-to object is destructed.
    - Because the underlying object's type in unique pointers does not matter, the unique pointer is a prime candidate for a template class.


=Type Checking in Templates=

Templates are type safe!
    - During template instantiation, template parameters are pasted in by the compiler.
    - The compiler will not generate the instantiation if the resulting code is incorrect.

Pointers don't support multiplication!

Similarities with duck-typed languages:
    - In duck-type languages such as Python, type checking is deferred until runtime.

What is duck-typed language?
    - If an object looks like a duck and quacks like a duck, then it must be some type of duck.

Implications:
    - This means that you will not know whether an object supports a particular operation until the program is executed.

One cannot know whether an instantiation will succeed until one tries to compile it.

Duck-typed languages might blow up at runtime.
Templates might blow up at compile time.

C++ programmers feel this is unacceptable, and has resulted in a solution to this problem called "Concepts".


=Concepts=

What do "Concepts" do?
    - Constrain template parameters.
    - Allow for parameter checking at the point of instantiation, rather than the point of first use.

Why is this good?
    - Catching usage issues at the point of instantiation allows the compiler to give an informative error code.
    - Allows the programmer to express requirements on template parameters directly in the language.

Concepts are officially part of the C++ language as part of the C++20 Standard.

Introduced to improve template programming, concepts provide a way to specify requirements on template arguments using compile-time Boolean predicates.  They were formally standardized in C++20 after years of development and revisions, including the removal of an earlier version (C++0x concepts) due to complexity.

(As of the writing of CP Crash Course, Concepts are NOT officially part of the C++ language!)

    -Defining a Concept-

        What is a Concept?
        - A Concept is a template.
        - A constant expression involving template arguments, evaluated at compile time.
        - A function that evaluates to true or false.

        How do Concepts work?
        - A set of template parameters must meet the criteria for a given concept.
        - If the Concept evaluates to true when instantiated with those parameters, the template instantiation succeeds.
        - If the Concept evaluates to false, the the template instantiation fails.

        Declaration:
        - Concepts are declared using the keyword, "concept".

        ```
        template <typename T1, typename T2, ...>
        concept bool ConceptName() {
            --snip--
        }
        ```

    -Type Traits-
    Concepts validate type parametrs.
    Types are manipulated withing Concepts for inspection of their properties.

    Two choices:
        1. These manipulations can be hand rolled.
        2. The type support library built into stdlib can be used.
    
    <type_traits> header:
    If one chooses to use the utilities in stdlib, these utilities are called, "Type Traits". They are part of the std namespace.

    Each type trait is a template class that takes a single template parameter (the one to be inspected).

    Results are extracgted using the template's static member "value".

    If the "value" member equals "true", then the type parameter meets the criteria. Otherwise, it will be qual to "false".

    Listing 6-18 demonstrates the investigation of type traits with several types.

    What are type traits good for?
        - Forming the building blocks of a Concept.
        - They can tell one what types are.

    Sometimes one must also specify how the template will use the types. The solution to this problem is the use of "Requirements".

    -Requirements-

    What are Requirements?
        - Ad hoc constraints on template parameters.
        - A Concept can specify any number of Requirements on its template parameters.
        - Requirements are encoded into requires expressions denoted by the "requires" keyword, followed by function arguments and a body.
    
    What is a Requirement composed of?
        - A sequence of syntactic requirements comprises the Requirements expression's body.
        - Each syntactic requirement puts a constraint on the template parameters.
    
    ```
    requires (arg-1, arg-2, ...) {
        {expression1} -> return-type1;
        {expression2} -> return-type2;
        --snip--
    }
    ```

    The arguments have types derived from template parameters.
    An arbitrary expression is placed within each of the braces.
    The expression can involve any number of the arguments to the argument expression.

    Failure:
        - If an instantiation causes a syntactic expression not to compile, that syntactic requirement fails.
    
    Success:
        - If the expression evaluation succeeds, the next check is whether the return type of that expression matches the type given after the arrow ->.
        
        If the expression result's evaluated type can't implicitly convert to the return type:
            - The syntactic requirement fails.
        
        If any of the syntactic requirements fail:
            - The Requires expression evaluates to "false".
        
        If all of the syntactic requirements pass:
            - The Requires expression evaluates to "true".
    
    Determine whether one can compare objects of types T and U using the equality and inequality operators:

    ```
    // T, U are types:
    requires (T t, U u) {
        {t==u} -> bool; // syntactic requirement 1
        {u==t} -> bool; // syntactic requirement 2
        {t!=u} -> bool; // syntactic requirement 3
        {u!=t} -> bool; // syntactic requirement 4
    }
    ```

    Explanation:
        - The Requires expression takes two arguments.
        - All four requirements enforce a bool result.
        - Any two types that satisfy this Requires expression are guaranteed to support comparison with == and !=.

    -Building Concepts from Requires Expressions-

    Requires expressions are evaluated at compile time, so Concepts can contain any number of them.

    -Using Concepts-

    After defining a Concept, use that concept's name in place of the typename keyword in a template declaration.

    When you try to use it with a type that is not compatible, the compiler error mssage will be much better than from a raw template.

    (Check current C++ standards for impleimentation of Concepts!)

    Whenever a Concept is used as apart of a function definition, that function becomes a template, and the template prefix can be omitted.

    ```
    return-type function-name(Concept arg-1, ...) {
        --snip--
    }
    ```

    -Ad Hoc Requires Expressions-

    Requirement can be enforced directly in the template prefix.

    Requires expressions can be embeded directly into the template definition.

=static_assert: The Preconcepts Stopgap=

Before the implementation of Concepts, there is a stopgap: The static_assert exression. Knowledge of this may be useful as Concepts are not necessarily available across compilers, especially older ones.

These combined with type_traits can be included in the bodies of templates to assist users in diagnosing usage errors.


=Non-Type Template Parameters=

A template parameter declared with the typename or class keyword is called a "type template parameter". It is a stand-in for some yet-to-be-specified type.

Non-type template parameters can also be used as stand-ins for a yet-to-be-specified value.

They can be any of the following:
    - integral
    - lvalue ref
    - pointer (or pointed-to member)
    - std::nullptr_t (the type of nullptr)
    - enum class

This allows you to inject a value into the generic code at compile time.


=Variadic Templates=

Allows you to write templates that must take in an unknown number of arguments. This is to avoid having to write multiple different templates each for different numebrs of arguments.


=Advanced Template Topics=

    -Template Specialization-
    Different versions of templates can be created depending on the characteristics of the parameters. If a type parameter is Ordered instead of Regular, you might be able to make a generic program more efficient. This type of programming is called "template specialization".

    The ISO standard temp.spec should be referred to for more information.

    -Name Binding-
    Helps determine the rules for when the compiler matches a named element withing a template to a concrete implementation.

    -Type Function-
    Takes types as arguments and returns a type.

    Type functions can be combined with compile time control structures to do general computation, such as programming control flow at compile time. Using these techniques is called, "template metaprogramming".

    -Template Metaprogramming-
    Easier once Concepts are part of the C++ standard.

=Template Source Code Organization=

Templates should be implemented entirely within header files.

Advantages to this approach:
    - Easy for others to use your code.
    - Easy for compilers to inline header-only templates, leads to faster code at runtime.
    - Compilers can do a better job of optimizing code when all of the source is available.


=Polymorphism at Runtime vs. Compile Time=

Templates should be used to achieve polymorphism. Sometimes this cannot be done because you don't know the types used with your code until runtime.

Template instantiation only occurs when a template's parameters are paired with types. The compiler can instantiate a custom class for you.

In such cases, use runtime polymorphism. The template is the mechanism for achieving compile-time polymorphism- runtime polymorphism is the interface.
